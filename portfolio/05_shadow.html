CS250 Shadow Mapping

WebGL2 / C++ / Emscripten

Project Overview

Shadow Mapping is a CS250 Computer Graphics assignment that implements real-time shadow rendering using a two-pass rendering technique. In the first pass, the scene is rendered from the light's point of view into a depth texture (shadow map). In the second pass, each fragment's position is transformed into light space and compared against the stored depth to determine whether it is in shadow.

The demo supports multiple object models (sphere, torus, trefoil knot, cylinder, cone, cube, and plane), adjustable light position and projection parameters, Percentage Closer Filtering (PCF) for soft shadow edges, fog, and a wireframe visualization of the light's view frustum. A depth map preview panel lets you inspect the raw shadow map in real time.

C++ WebGL2 GLSL ES 3.0 Emscripten OpenGL ES 3.0 ImGui CMake FBO

Tasks Completed

Depth Map Rendering

  • Rendered the scene from the light's POV into a depth-only FBO
  • Supports both orthographic (directional) and perspective (spot) light projections
  • Depth texture bound as sampler2DShadow for hardware comparison

PCF Soft Shadows

  • Implemented Percentage Closer Filtering with a configurable kernel size
  • Used textureProj with sampler2DShadow for per-sample depth comparison
  • Produces smooth shadow penumbrae without blurring the depth map itself

Procedural Geometry

  • Generated sphere, torus, trefoil knot, cylinder, cone, cube, and plane meshes procedurally
  • All meshes share a unified vertex layout (position, normal, UV) for seamless shader reuse
  • Correct outward face normals computed from parametric surface cross products

Light Frustum Visualization

  • Rendered the light's view frustum as a wireframe box in world space
  • Built from 12 explicit edge pairs to avoid diagonal artifacts from triangle decomposition
  • Transformed from NDC to world space using the inverse light view-projection matrix

Challenge & Lesson

Challenge — VAO EBO Corruption Crash

The app crashed immediately on startup with a STATUS_ACCESS_VIOLATION. The root cause was that SetIndexBuffer left the VAO bound after attaching the index buffer. The next IndexBuffer constructor then bound a new EBO to the still-active VAO and unbound it, silently zeroing out the previous VAO's stored EBO handle. When glDrawElements later tried to read from EBO=0 it dereferenced address 0 and crashed. Adding a single glBindVertexArray(0) at the end of SetIndexBuffer fixed the crash.

Challenge — Inside-Out Sphere Normals

The sphere, torus, and trefoil knot all appeared hollow — back-face culling was removing the outer surface. The parametric grid generates quads where one axis (phi / major circle / tangent) points "downward" in parameter space, making the default {p00, p01, p10} winding produce inward-facing normals. Swapping to {p00, p10, p01} reversed the winding so the cross product points outward, making the surfaces render correctly.

Challenge — Wireframe Diagonal Lines

The light frustum wireframe displayed diagonal lines across each face because to_submesh_as_lines converts triangle meshes to line pairs — including the shared diagonal edge that splits each quad into two triangles. The fix was to bypass that helper and build the wireframe directly from 12 explicit cube edge pairs, producing a clean box with no diagonals.

Lesson — Two-Pass Rendering

Shadow mapping requires two full scene renders per frame: once from the light to build the depth map, and once from the camera to shade with shadows. Keeping the depth-pass shader minimal (position-only, no fragment output) is critical for performance since the depth map render is purely for geometry, not color. The shadow matrix — light projection × view × bias — maps world positions directly into the [0,1] depth texture space.
